# 编写自动化测试

在他 1972 年的文章《谦逊的程序员》中，Edsger W. Dijkstra 说过：“程序测试可以非常有效地表明 bug 的存在，但要证明 bug 不存在，却远远不够。”这并不意味着我们不应该尽可能多地进行测试！

我们程序的_正确性_，指的是代码执行的结果与我们的意图相符的程度。Rust 的设计高度重视程序的正确性，但正确性很复杂，也很难证明。Rust 的类型系统承担了其中很大一部分负担，但类型系统无法捕获所有问题。因此，Rust 内置了对编写自动化软件测试的支持。

假设我们编写了一个函数 `add_two`，它会将传入的任意数字加 2。这个函数的签名接收一个整数作为参数，并返回一个整数结果。实现并编译这个函数时，Rust 会执行你目前学到的所有类型检查和借用检查，以确保例如我们没有将 `String` 值或无效引用传递给该函数。但是，Rust _无法_ 检查这个函数是否会完全按照我们的意图执行：返回参数加 2 的结果，而不是比如参数加 10 或减 50！这正是测试发挥作用的地方。

我们可以编写测试来断言，例如，将 `3` 传给 `add_two` 函数时，返回值是 `5`。每次修改代码时，我们都可以运行这些测试，以确保已有的正确行为没有发生变化。

测试是一项复杂的技能：虽然不可能在一章中涵盖编写优秀测试的每个细节，但本章将讨论 Rust 测试功能的运作机制。我们会介绍编写测试时可用的注解和宏、运行测试时提供的默认行为和选项，以及如何将测试组织为单元测试和集成测试。
